# 컨텍스트 스위칭 (Context Switching)
## **컨텍스트 스위칭이란?**

### **정의**

운영체제 커널이 CPU 제어권을 하나의 프로세스(또는 스레드)에서 다른 것으로 넘기는 과정

쉽게 말해, CPU가 하던 일을 잠깐 멈추고 다른 일을 하기 위해 어디까지 했는지를 적어두고 새로운 일은 어디부터 하면 되는지를 읽어오는 작업

### **목적 : `멀티태스킹 (Multitasking)`**

CPU가 여러개의 프로세스들을 아주 짧은 시간 단위로 번갈아 가며 실행(컨텍스트 스위칭)하여 마치 동시에 실행되는 것처럼 보이게 만듦.

### **발생 시점**

- 타임 슬라이스 만료
- I/O 요청으로 인한 대기
- 인터럽트 발생
- 시스템 콜 호출

### **상태 저장 위치 : PCB**

프로세스의 정보(문맥)를 저장할 공간이 필요한데, 이때 사용되는 것이 **PCB(Process Control Block, 프로세스 제어 블록)**

- **PCB란**
    - 운영체제가 프로세스를 관리하기 위해 프로세스마다 유지하는 데이터 블록
    - 일종의 프로세스 주민등록증
- **저장되는 주요 정보**
    - 현재 실행 위치(Program Counter), CPU 레지스터 값, 프로세스 상태(Ready, Running 등), 메모리 관리 정보 등

---

## **컨텍스트 스위칭 과정**

```
┌─────────────────────────────────────────────────────┐
│                    Process A 실행 중                 │
└─────────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────┐
│  1. Process A의 상태(컨텍스트)를 PCB에 저장          │
│     - PC (Program Counter)                          │
│     - CPU 레지스터                                   │
│     - 메모리 관리 정보                               │
│     - I/O 상태 정보                                  │
└─────────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────┐
│  2. Process B의 상태를 PCB에서 복원                  │
└─────────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────┐
│                    Process B 실행 중                 │
└─────────────────────────────────────────────────────┘
```

1. **인터럽트 발생**
    
    현재 실행 중인 프로세스 A의 할당 시간이 끝나거나, 입출력(I/O) 요청이 발생하여 인터럽트가 걸림
    
2. **상태 저장**
    
    프로세스 A의 현재 상태(어디까지 실행했는지 등)를 프로세스 A의 PCB에 저장
    
3. **상태 복원**
    
    다음에 실행할 프로세스 B의 PCB에서 이전 상태를 읽어와 CPU 레지스터에 적재(복원)
    
4. **실행**
    
    프로세스 B가 이어서 실행

## **프로세스 vs 스레드 컨텍스트 스위칭**

- **프로세스 스위칭**
    - 프로세스는 독립적인 메모리 공간을 가지므로, 스위칭할 때 메모리 주소 체계(가상 메모리 등)까지 전부 바꿔야 해서 무겁고 느림
    - 캐시 메모리도 완전히 비워야 함
- **스레드 스위칭**
    - 같은 프로세스 내의 스레드들은 코드, 데이터, 힙 영역을 공유
    - 따라서 스위칭할 때 공유하는 영역은 그대로 두고, 스레드 고유의 영역인 PC(Program Counter)와 레지스터, 스택 영역만 교체하면 되므로 훨씬 가볍고 빠름

### **프로세스 간 스위칭 (비용 높음)**

| **작업** | **설명** |
| --- | --- |
| 메모리 주소 공간 전환 | 페이지 테이블 교체 필요 |
| TLB 플러시 | 가상 주소 → 물리 주소 캐시 무효화 |
| CPU 캐시 미스 | L1, L2, L3 캐시 적중률 저하 |

### **스레드 간 스위칭 (비용 낮음)**

| **작업** | **설명** |
| --- | --- |
| 레지스터만 교체 | Stack 포인터, PC 등 |
| 메모리 공유 | TLB 플러시 불필요 |
| 캐시 유지 | 공유 메모리로 캐시 적중률 유지 |

---

## **주의해야할 점 : 오버헤드 (Overhead)**

- 컨텍스트 스위칭이 일어나는 동안 CPU는 실질적인 데이터 처리(유용한 작업)를 전혀 하지 못함.
- 너무 잦은 컨텍스트 스위칭은 오히려 시스템 전체의 성능을 떨어뜨리는 원인
- 스위칭 시 기존 CPU 캐시 메모리에 있던 데이터가 쓸모없어지고 새로 캐시를 채워야 하는 '캐시 미스(Cache Miss)' 비용도 발생

---

## **최적화 전략**

- **스레드 풀 사용**: 스레드 생성/소멸 비용 절감
- **비동기 I/O**: 블로킹으로 인한 스위칭 최소화
- **CPU 친화도 설정**: 특정 코어에 스레드 바인딩
- **캐시 효율성 고려**: 데이터 지역성 활용
